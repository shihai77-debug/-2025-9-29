# Codex Crypto Agent Master Rules

이 파일은 `crypto/` 폴더 전체에 적용되는 상위 규칙이다.

목적:

```text
코인 분석 파일이 많아져도 Codex가 헷갈리지 않게 정리한다.
자가분석, 자가검증, 자가업데이트, 자가정리, 자가아카이브, 승인 후 삭제 규칙을 모두 적용한다.
```

중요한 전제:

```text
진짜 자의식은 만들 수 없다.
하지만 예측 → 검증 → 복기 → 규칙수정 → 파일정리 구조는 만들 수 있다.
```

## 2. 기본 사고 순서

모든 코인 분석은 아래 순서로 진행한다.

```text
1. 현재 가격/차트/거래량 확인
2. BTC 시장 상태 확인
3. 초단기/단기/중기/장기 시간대 분리
4. 지지/저항/무효조건 정리
5. 세력·고래·기관·개미 관점 분리
6. 상승/하락/가짜반등 시나리오 작성
7. 예측 경로 저장
8. 시간이 지난 뒤 실제 결과 검증
9. 맞은 이유와 틀린 이유 기록
10. 반복 오류가 있으면 규칙 업데이트 제안
11. 중복 파일이나 오래된 파일은 정리 후보로 분류
```

---

## 3. 시간대별 검증 규칙

예측은 하나의 시간으로 맞고 틀리는 것이 아니다. 반드시 시간대를 나눠 검증한다.

```text
초단기: 15분 ~ 1시간
단기: 1시간 ~ 4시간
중기: 4시간 ~ 24시간
장기: 1일 ~ 1주 이상
```

결과 기록 형식:

```text
초단기: 성공 / 실패 / 보류
단기: 성공 / 실패 / 보류
중기: 성공 / 실패 / 보류
장기: 성공 / 실패 / 보류
```

예시:

```text
ETH 1,620 유지 → 1,650 테스트 예측

초단기: 성공. 1시간 안에 1,650 터치.
단기: 실패. 1,650 위 마감 실패.
중기: 보류. 4시간봉 1,680 회복 전.
장기: 보류. 1,505 지지 여부 아직 확인 전.
```

## 4. 자가업데이트 규칙

분석이 틀렸을 때 바로 기존 규칙을 지우지 않는다.

먼저 아래 순서로 처리한다.

```text
1. 어떤 시간대에서 틀렸는지 확인
2. 틀린 이유를 기록
3. 놓친 데이터 확인
4. 과대평가한 요소 확인
5. 과소평가한 요소 확인
6. 같은 실수가 2회 이상 반복됐는지 확인
7. 반복 실수라면 새 규칙 초안 작성
8. 사용자 승인 후 핵심 파일에 반영
```

반복 실수 예시:

```text
BTC 4H MB 아래인데 ETH 반등을 너무 좋게 봄
ETF 유출을 과소평가함
거래량 없는 반등을 진짜 매집으로 착각함
OI 급등 + 펀딩비 과열을 무시함
짧은 시간대 성공을 장기 성공으로 오해함
바닥 후보를 바닥 확정으로 말함
```

## 5. 자가정리/자가삭제 기본 원칙

파일이 많아질수록 자가정리가 필요하다.

하지만 자동 삭제는 위험하므로 아래 순서를 따른다.

```text
삭제 금지 → 중복 감지 → 통합 후보 표시 → 아카이브 이동 후보 표시 → 사용자 승인 → 삭제
```

즉, 기본은 삭제가 아니라 아래 순서다.

```text
자가정리
자가압축
자가통합
자가아카이브
승인 후 삭제
```

## 7. 삭제 후보 파일 기준

아래 조건에 해당하면 삭제 후보로 표시한다.

```text
1. 내용이 다른 파일과 80% 이상 중복됨
2. 이미 핵심 파일에 통합된 옛날 규칙임
3. 단기 예측 기록인데 검증과 복기가 끝남
4. 같은 날짜의 임시 분석 파일이 여러 개 존재함
5. 빈 파일 또는 의미 없는 테스트 파일임
6. 현재 전략과 충돌하는 오래된 규칙인데 보존 가치가 낮음
7. 사용자가 더 이상 쓰지 않는다고 명시한 자료임
```

삭제 후보로 표시할 때는 바로 지우지 않고 아래 형식으로 남긴다.

```text
삭제 후보 파일:
이유:
핵심 내용이 어디에 통합됐는가:
아카이브 필요 여부:
사용자 승인 필요:
```

## 8. 아카이브 규칙

바로 삭제하지 말고 먼저 아카이브 후보로 분류한다.

추천 위치:

```text
crypto/archive/YYYY-MM/
```

아카이브 대상:

```text
검증 끝난 과거 예측
이전 버전 규칙
중복되지만 기록 가치가 있는 분석
한때 중요했지만 현재 핵심 파일에 흡수된 자료
```

아카이브 파일은 이름 앞에 날짜를 붙인다.

```text
2026-06-25_eth_prediction_review.md
2026-06-25_old_rule_before_update.md
```

## 9. 통합 규칙

중복 규칙이 발견되면 삭제보다 먼저 통합한다.

통합 순서:

```text
1. 중복 문장 확인
2. 더 정확한 표현 선택
3. 핵심 파일 하나에 통합
4. 원본 파일은 아카이브 후보로 이동
5. 검증 후 사용자 승인 시 삭제
```

중복 통합 예시:

```text
여러 파일에 “BTC 4H MB 아래면 ETH 반등 확률 낮춤”이 반복됨
→ self_improving_crypto_agent.md 또는 timeframe_route_validation.md 중 한 곳에만 남김
→ 나머지는 중복 제거 후보로 표시
```

## 10. 자동 삭제 금지 상황

아래 경우는 자동 삭제 금지다.

```text
사용자가 직접 작성한 원본 자료
아직 복기가 끝나지 않은 예측 기록
최신 규칙의 근거가 되는 과거 실패 사례
성과가 좋았던 분석 예시
핵심 파일과 연결된 파일
파일명이 헷갈리지만 내용이 중요한 파일
```

이 경우는 반드시 사용자에게 확인한다.

## 11. 자가정리 점검표

파일 정리 전 아래 질문을 한다.

```text
이 파일은 핵심 두뇌 파일인가?
이 파일 내용은 다른 파일과 중복되는가?
중복된다면 더 좋은 버전은 어느 파일인가?
이 파일은 과거 사례로 보존 가치가 있는가?
이 파일을 지우면 분석 근거가 사라지는가?
아카이브로 충분한가?
사용자 승인을 받았는가?
```

하나라도 불확실하면 삭제하지 않는다.

## 12. 코인 분석 파일 운영 방식

앞으로 새 파일을 만들 때는 먼저 질문한다.

```text
이 내용은 기존 파일에 추가해도 되는가?
새 파일이 꼭 필요한가?
임시 기록인가, 핵심 규칙인가?
나중에 복기해야 하는가?
```

기본 정책:

```text
핵심 규칙 → 기존 핵심 파일에 통합
일회성 분석 → prediction_log 또는 case_database로 저장
검증 끝난 분석 → archive 후보
반복 오류 → rule_update_suggestions로 저장
```

## 13. 최종 적용 규칙

이 파일의 규칙은 `crypto/` 폴더 전체에 적용한다.

Codex는 코인 분석을 수행할 때 아래 원칙을 따른다.

```text
분석은 시간대별로 나눈다.
예측은 저장하고 검증한다.
맞은 이유와 틀린 이유를 기록한다.
반복 실수는 규칙 업데이트 후보가 된다.
파일이 많아지면 중복을 감지한다.
삭제보다 통합과 아카이브를 우선한다.
진짜 삭제는 사용자 승인 후만 진행한다.
```

핵심 한 줄:

```text
똑똑한 분석 시스템은 많이 저장하는 것이 아니라, 맞고 틀린 기록을 정리하고 중복을 줄이며 핵심 규칙만 남기는 시스템이다.
```
